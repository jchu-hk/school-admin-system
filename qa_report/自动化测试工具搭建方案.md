## 1. 自动化测试策略

### 1.1 测试金字塔模型

```
                    ▲
                   /  \
                  / E2E \          20% - UI自动化 (Playwright)
                 /________\
                /          \
               / Integration \    30% - 接口自动化 (API Test)
              /________________\
             /                  \
            /     Unit Test       \  50% - 单元测试 (Jest/Vitest)
           /________________________\

执行速度: 快 ────────────────────────────> 慢
维护成本: 低 ────────────────────────────> 高
稳定性:   高 ────────────────────────────> 低
```

### 1.2 自动化测试范围

| 测试层级 | 自动化策略 | 覆盖功能 | 执行频率 | 工具选择 |
|---------|-----------|---------|---------|---------|
| **单元测试** | 开发自测，CI必跑 | 核心业务逻辑、工具函数 | 每次提交 | Jest/Vitest |
| **接口测试** | 契约测试，Mock外部依赖 | API端点、数据验证 | 每次提交 | Postman/Newman + Supertest |
| **集成测试** | 数据库+缓存集成 | 数据流、事务一致性 | 每日构建 | Playwright + TestContainers |
| **E2E测试** | 核心业务流程 | P0功能端到端场景 | 每日构建 | Playwright |
| **视觉回归** | 关键页面截图对比 | UI组件、报表页面 | 每周 | Playwright + Argos CI |
| **性能测试** | 基准测试 | API响应时间 | 每周 | k6 |
| **安全测试** | 漏洞扫描 | OWASP Top 10 | 每周 | OWASP ZAP |

### 1.3 自动化优先级矩阵

| 功能模块 | 优先级 | 自动化类型 | 用例数量 | 原因 |
|---------|--------|-----------|---------|------|
| 用户认证/授权 | P0 | E2E + 接口 | 15 | 核心安全路径 |
| 出勤管理 | P0 | E2E + 接口 | 12 | 高频使用，数据敏感 |
| 收费/财务 | P0 | E2E + 接口 | 10 | 涉及金钱，需精确 |
| 请假/审批流 | P0 | E2E | 8 | 流程复杂，多状态 |
| 成绩单/DSE | P1 | 接口 + 抽样E2E | 8 | 周期性使用 |
| AI/OCR功能 | P1 | 接口 + 抽样E2E | 6 | 依赖AI服务，需Mock |
| 报表/统计 | P2 | 接口 | 5 | 数据查询为主 |
| 系统配置 | P2 | 接口 | 4 | 管理员操作 |

## 3. 推荐方案

### 3.1 最终推荐: Playwright

基于以上多维度分析，**强烈推荐使用 Playwright** 作为智能校务助理系统的自动化测试框架。

#### 推荐理由

1. **最高综合评分**: 9.2分，领先Cypress 1.5分，领先Selenium 3.6分
2. **维护成本最低**: 自动等待机制大幅减少flaky test
3. **浏览器覆盖最全**: 唯一原生支持WebKit(Safari)的框架
4. **团队技能匹配**: Node.js/TypeScript技术栈与项目一致
5. **CI集成友好**: 免费并行执行，原生Docker支持
6. **Microsoft背书**: 活跃开发，社区增长最快

### 4.3 核心配置文件

#### playwright.config.ts

```typescript
import { defineConfig, devices } from '@playwright/test';
import { testConfig } from './config/test-config';

/**
 * Playwright 配置文件
 * @see https://playwright.dev/docs/test-configuration
 */
export default defineConfig({
  // 测试目录
  testDir: './tests',
  
  // 每个测试文件完全并行
  fullyParallel: true,
  
  // 失败时禁止并行(便于调试)
  workers: process.env.CI ? 3 : undefined,
  
  // 重试策略: CI环境重试2次，本地不重试
  retries: process.env.CI ? 2 : 0,
  
  // 每个测试超时时间
  timeout: 60 * 1000, // 60秒
  
  // 期望条件超时
  expect: {
    timeout: 10 * 1000, // 10秒
  },
  
  // 全局setup/teardown
  globalSetup: require.resolve('./utils/global-setup'),
  globalTeardown: require.resolve('./utils/global-teardown'),
  
  // 测试报告配置
  reporter: [
    // 终端列表报告
    ['list'],
    // HTML报告
    ['html', { 
      outputFolder: 'reports/html',
      open: process.env.CI ? 'never' : 'on-failure',
    }],
    // JUnit XML报告(CI用)
    ['junit', { 
      outputFile: 'reports/junit/results.xml',
    }],
    // Allure报告
    ['allure-playwright', {
      detail: true,
      outputFolder: 'reports/allure',
      suiteTitle: false,
    }],
  ],
  
  // 共享配置
  use: {
    // 基础URL
    baseURL: process.env.BASE_URL || 'http://localhost:3000',
    
    // 浏览器上下文选项
    headless: process.env.CI ? true : false,
    
    // 跟踪配置: 仅在失败时保留
```

#### pages/BasePage.ts

```typescript
import { Page, Locator, expect } from '@playwright/test';

/**
 * 基础页面类
 * 所有Page Object的基类
 */
export abstract class BasePage {
  readonly page: Page;
  readonly url: string;

  constructor(page: Page, url: string = '/') {
    this.page = page;
    this.url = url;
  }

  /**
   * 导航到页面
   */
  async goto(): Promise<void> {
    await this.page.goto(this.url);
    await this.waitForLoad();
  }

  /**
   * 等待页面加载完成
   */
  async waitForLoad(): Promise<void> {
    await this.page.waitForLoadState('networkidle');
  }

  /**
   * 获取元素文本
   */
  async getText(locator: Locator): Promise<string> {
    return await locator.textContent() || '';
  }

  /**
   * 安全点击(带重试)
   */
  async safeClick(locator: Locator): Promise<void> {
    await locator.waitFor({ state: 'visible' });
    await locator.click();
  }

  /**
   * 填写表单并验证
   */
  async fillAndVerify(locator: Locator, value: string): Promise<void> {
    await locator.fill(value);
    await expect(locator).toHaveValue(value);
  }

  /**
   * 等待toast消息并验证
   */
  async expectToast(message: string): Promise<void> {
    const toast = this.page.locator('[data-testid="toast-message"]');
    await expect(toast).toContainText(message);
    await toast.waitFor({ state: 'hidden', timeout: 5000 });
  }

  /**
   * 截图(带时间戳)
   */
  async screenshot(name: string): Promise<void> {
    const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
    await this.page.screenshot({ 
      path: `reports/screenshots/${name}-${timestamp}.png`,
      fullPage: true 
    });
  }
}
```

### 8.1 F-ENRL-003课本分发自动化测试

**测试目标：** 验证课本分发全流程，包括分发登记、费用结算、退换处理和汇总报表生成。

**用例编号：** ENRL-001-A1 ~ ENRL-001-A8（对应手工测试用例ENRL-003-01至ENRL-003-08）

**测试覆盖场景：**
- ✅ 班级课本分发清单生成（ENRL-001-A1）
- ✅ 全免资格学生费用豁免（ENRL-001-A2）
- ✅ 课本批次信息自动填充（ENRL-001-A3）
- ✅ FPS缴费与电子收据推送（ENRL-001-A4）
- ✅ 错发课本更换处理（ENRL-001-A5）
- ✅ 超时办理分发被拦截（ENRL-001-A6）
- ✅ 课本退货退款计算（ENRL-001-A7）
- ✅ 当日分发汇总报告生成（ENRL-001-A8）

**示例用例（ENRL-001-A1）：**

```typescript
test.describe('F-ENRL-003 课本分发管理自动化 @enrl @smoke', () => {
  test('ENRL-001-A1: 班级课本分发清单生成', async ({ page }) => {
    // Arrange: 登录并进入课本分发模块
    await loginAsOfficer(page);
    const textbookPage = new TextbookPage(page);
    await textbookPage.goto();
    
    // Act: 选择班级和学年
    await textbookPage.selectClass('1A');
    await textbookPage.selectAcademicYear('2026-2027');
    await textbookPage.clickGenerateList();
    
    // Assert: 验证清单生成
    await expect(page.getByTestId('student-list')).toBeVisible();
    const studentCount = await page.getByTestId('student-list').locator('tr').count();

  });
});
```

**测试目标：** 验证敏感权限变更审批时的二次认证流程，包括OTP、扫码和硬件Token验证。

**测试覆盖场景：**

- ✅ 扫码验证流程
- ✅ 硬件Token验证流程
- ✅ 验证失败重试机制
- ✅ 验证成功后权限生效

**示例用例（OTP验证）：**

```typescript
test.describe('二次认证弹窗自动化 @mfa @security', () => {
  test('OTP验证成功后权限生效', async ({ page }) => {
    // Arrange: 教师提交跨班级数据访问申请
    await loginAsTeacher(page);
    const permissionPage = new PermissionPage(page);
    await permissionPage.goto();
    await permissionPage.submitCrossClassAccessRequest(['2B', '2C'], '活动报名协助');
    
    // Act: 校务主任审批，触发OTP验证
    await loginAsAdmin(page);
    await permissionPage.navigateToApproval('ESC-2026-001');
    await page.getByTestId('approve-button').click();
    
    // 二次认证弹窗出现
    await expect(page.getByTestId('mfa-modal')).toBeVisible();
    
    // 输入测试OTP
    await page.getByTestId('otp-input').fill('123456');
    await page.getByTestId('verify-otp').click();
    
    // Assert: 验证成功，权限生效
    await expect(page.getByTestId('approval-success')).toContainText('审批通过');
    await permissionPage.verifyPermissionGranted();
  });
  
  test('OTP验证失败重试', async ({ page }) => {
    // Arrange: 触发OTP验证
    await loginAsAdmin(page);
    const permissionPage = new PermissionPage(page);
    await permissionPage.navigateToApproval('ESC-2026-001');
    await page.getByTestId('approve-button').click();
    
    // Act: 输入错误OTP
    await page.getByTestId('otp-input').fill('000000');
    await page.getByTestId('verify-otp').click();
    
    // Assert: 显示错误提示

    // 输入正确OTP
    await page.getByTestId('otp-input').fill('123456');

    // 验证成功
    await expect(page.getByTestId('approval-success')).toBeVisible();
  });
});
```

### 7.2 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 测试在CI中失败但本地通过 | 环境差异/时序问题 | 增加等待时间，使用docker-compose统一环境 |
| Flaky test增加 | 定位不稳定/数据竞争 | 使用更稳定的定位器(data-testid)，增加隔离 |
| 测试执行变慢 | 测试数据膨胀 | 定期清理测试数据，使用fixtures隔离 |
| 浏览器启动失败 | 浏览器版本不匹配 | 更新Playwright到最新版本 |
| 视觉回归误报 | UI微调 | 调整对比阈值，更新基线 |
