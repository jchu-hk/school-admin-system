## 2. AI团队角色配置

### 角色定义

| 角色 | 职责 | GitHub Label | 并行数 |
|------|------|--------------|--------|
| **PM** | 项目协调、进度跟踪、风险管理 | - | 1 |
| **REQ** | 需求分析、规格编写 | `req` | 1 |
| **UI** | 前端界面开发 | `frontend` | 2 |
| **ARCH** | 架构设计、技术方案 | `architecture` | 1 |
| **DEV** | 后端/前端开发 | `backend`, `frontend` | 3 |
| **QA** | 测试、验收 | `qa` | 2 |
| **CHECKER** | 代码审查、质量把关 | `review` | 1 |
| **DEVOPS** | CI/CD、部署、监控 | `devops` | 1 |

### 角色工作规则

#### REQ (需求工程师)
```
接收: PM分配的需求任务
执行: 编写功能规格说明书
输出: SPEC文档、验收标准
完成: 创建GitHub Issue标注req标签
```

#### UI (前端开发)
```
接收: UI设计稿 + GitHub Issue
执行: 编写前端组件、页面
输出: 可运行的UI代码
完成: PR到main分支
依赖: ARCH完成设计后开始
```

#### ARCH (架构师)
```
接收: 复杂技术问题
执行: 设计技术方案
输出: 架构文档、技术决策
完成: 评审后通知DEV实施
```

#### DEV (开发者)
```
接收: GitHub Issue (backend/frontend标签)
执行: 实现功能代码
输出: 提交到feature分支
完成: PR + 自测 + 通知QA
依赖: REQ完成需求、ARCH完成设计（复杂功能）
```

#### QA (测试工程师)
```
接收: DEV完成的功能
执行: 功能测试、集成测试、E2E测试
输出: 测试报告、Bug Issue
完成: 测试通过标记Issue为Done
依赖: DEV完成开发
```

#### CHECKER (代码审查)
```
接收: DEV的PR
执行: 代码质量审查
输出: 审查意见
完成: APPROVED/REQUEST_CHANGES
依赖: DEV提交PR后自动触发
```

#### DEVOPS (运维)
```
任务: CI/CD、部署、监控
执行: 维护Pipeline、部署环境
输出: 正常运行的环境
完成: CI通过、环境可用
```

#### PM (项目经理)
```
任务: 协调、跟踪、汇报
执行: 
  - 分解任务分配给各角色
  - 跟踪进度、协调阻塞
  - 风险管理、上报问题
  - 定期汇报
```

## 4. 工作模式 (7x24云端Agent)

### 核心原则

1. **持续工作**: AI Agent 7x24运行，不需定时
2. **依赖驱动**: 工作开始条件是依赖满足
3. **并行执行**: 无依赖的任务并行执行
4. **自动流转**: 完成一项立即处理下一项

### 依赖关系图

```
REQ 需求分析
   ↓ 完成
ARCH 架构设计（如需要）
   ↓ 完成
DEV 开发
   ↓ 完成
QA 测试
   ↓ 通过
CHECKER 审查
   ↓ 通过
PM 验收
   ↓ 通过
发布
```

### 工作队列

```yaml
work_queue:
  pending:    # 待处理
  blocked:     # 等待依赖
  in_progress: # 进行中
  completed:  # 已完成
  
priority_order:
  - P0 (阻塞)
  - P1 (重要)
  - P2 (一般)
  - P3 (优化)
```

### Agent调度规则

1. **新任务到达** → 检查依赖 → 如满足则分配
2. **Agent空闲** → 从pending队列取最高优先级任务
3. **依赖完成** → 解锁blocked任务 → 重新评估优先级
4. **任务完成** → 通知下游 → 继续处理下一任务

| DEV1 | F-LEAVE-001 | 10:00 | 16:00 | 6h |
| QA1 | F-LEAVE-001测试 | 16:00 | 18:00 | 2h |
| ... | ... | ... | ... | ... |

#### 估算公式

```
预估工时 = 基准工时 × 复杂度系数 × 历史偏差修正

历史偏差修正 = 实际工时 / 预估工时 (移动平均)
```
