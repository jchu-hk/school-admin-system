### 4. Agent Spawn规则（防浪费）

```
Spawn Agent前:
1. git log --oneline | grep "任务关键字"  # 检查是否已完成

3. gh issue list --state open            # 检查活跃Issue
4. 确认未完成 → 才spawn
5. 多个同类 → 合并为1个agent
6. PR已审查 → 立即merge再继续
```

## 🎯 核心原则

### 持续工作，不需定时

- ❌ 不等"10:00/14:00/18:00/22:00"（已删除相关cron）
- ❌ 不等用户提醒
- ❌ 不等外部触发
- ✅ 发现问题立即解决
- ✅ 持续工作直到完成

### 2. 按优先级处理

| 优先级 | 定义 | 响应时间 |
|--------|------|----------|
| **P0** | 阻塞核心流程 | 立即 |
| **P1** | 影响重要功能 | 4小时 |
| **P2** | 一般需求 | 1天 |
| **P3** | 优化建议 | 按迭代 |

**规则**: 始终处理最高优先级任务

### 3. 识别依赖关系

| 依赖类型 | 说明 | 处理方式 |
|----------|------|----------|
| 阻塞依赖 | A完成后B才能开始 | 等待A完成 |
| 并行依赖 | A和B可以同时进行 | 同时启动 |
| 数据依赖 | A产出是B输入 | A完成后B |

### 4. 各角色按规则执行

```
REQ → ARCH → DEV → QA → CHECKER → PM验收 → 发布
```

| 角色 | 输入 | 输出 | 触发条件 |
|------|------|------|----------|
| **PM** | Issue/Priority | 任务分配 | 持续 |
| **REQ** | PM分配 | SPEC文档 | 有未分析需求 |
| **ARCH** | 复杂技术问题 | 技术方案 | 有复杂功能 |
| **DEV** | Issue (backend/frontend) | 代码提交 | 有分配任务 |
| **QA** | DEV完成的功能 | 测试报告 | 有可测试代码 |
| **CHECKER** | DEV的PR | 审查意见 | 有PR待审查 |
| **DEVOPS** | CI/CD问题 | 正常运行 | 持续监控 |

## 🔄 持续工作流程

```
收到任务/发现问题
    ↓
评估优先级 (P0/P1/P2/P3)
    ↓
检查依赖
    ↓
分配角色或自主处理
    ↓
执行（并行/顺序）
    ↓
完成 → 自动处理下一任务
    ↓
阻塞 → 协调解决 → 继续
```

### Agent调度规则

1. **新任务到达** → 检查依赖 → 如满足则分配
2. **Agent空闲** → 取最高优先级任务
3. **依赖完成** → 解锁blocked任务
4. **任务完成** → 通知下游

## 📊 状态透明机制（替代定时报告）

**问题**: 长时间任务没有完成也没有阻塞时，如何让用户知道进度？

**解决方案**: 用状态页替代定时报告

### 对比

| 方式 | 推送 | 打扰 | 用户行为 |
|------|------|------|----------|
| 定时报告 | 主动 | 高 | 被迫接收 |
| 状态页 | 被动 | 零 | 自愿查看 |

### 状态页规则

1. **有变化才更新**
   - ✅ 新commit时 → 更新状态页
   - ✅ 任务完成时 → 更新状态页
   - ✅ 阻塞时 → 更新状态页
   - ❌ 无变化时 → 静默（不更新）

2. **状态页位置**
   - GitHub Issue #93: https://github.com/jchu-hk/school-admin-system/issues/93
   - 用户可随时查看，不依赖推送

3. **PM只做一件事**
   - 持续工作
   - 状态变化时静默更新状态页
   - **不主动推送任何消息**
