## 第一部分：执行摘要

### 1.1 模拟目的

本报告通过角色扮演方式，模拟智能校务助理系统所有四类用户角色（校务主任、校务处同工、教师、家长/学生）在典型工作日的完整系统使用流程，从功能规格说明书（SPEC-COMPLETE.md v1.4.0）和系统设计文档出发，还原真实使用场景，收集可用性、功能性、合规性及用户体验层面的反馈。

### 1.2 模拟方法

| 项目 | 说明 |
|------|------|
| **模拟框架** | "Day-in-the-Life" (DITL) 场景模拟 |

| **模拟时间线** | 覆盖每日、每周、每月、每学期、每学年各周期 |
| **数据基础** | SPEC-COMPLETE.md (v1.4.0)、SPEC-SYSTEM-DESIGN.md、SPEC-UI-PROTO.md |
| **反馈收集** | 结构化场景后置反馈问卷（每场景5维度）|

### 1.3 核心发现摘要

| 类别 | 总数 | 高优先 | 中优先 | 低优先 |
|------|------|--------|--------|--------|
| 功能缺陷/缺失 | 12 | 4 | 5 | 3 |

### 1.4 高优先级改进建议（Top 5）

1. **[高] 家长门户消息触达率保障** — 微信渠道token状态显示"unknown"，通知可靠性无法保证
2. **[高] eClass同步失败时的完整人工兜底** — 部分同步失败需明确告知受影响学生列表
3. **[高] 零用现金双人见证流程在线协同** — 见证人需实时收到待见证任务推送
4. **[高] 敏感字段界面脱敏策略** — 身份证号、完整联系方式默认脱敏，二次确认才展示
5. **[高] 中一自招面试评分防作弊** — 评分需增加计时、版本控制和提交后锁定机制

### A.2 典型工作日：2026年5月27日（星期三）

#### 🕖 07:30 — 登录与晨间仪表板审阅

**场景描述：**
陈主任到校后第一件事是打开仪表板，查看当日整体运营状态。

**系统交互流程：**
```
[Step 1] 打开浏览器 → 输入 https://school-admin.school.edu.hk
[Step 2] 飞书扫码登录（F-USER-002 SSO认证）
[Step 3] 默认进入：每日晨检仪表板（F-DASH-001）
[Step 4] 查看六大组件状态
[Step 5] 点击「出勤率」→ 进入 F-ATT-001 出勤概览
[Step 6] 注意到「迟到/早退」status=warning → 点击查看详情
```

**系统输出（模拟）：**
```json
{
  "date": "2026-05-27", "weekday": "星期三",
  "components": [
    {"name": "出勤率", "value": "97.8%", "status": "normal"},
    {"name": "迟到/早退", "value": "8人", "status": "warning"},
    {"name": "家长查询", "value": "5个待处理", "status": "urgent"},
    {"name": "校车状态", "value": "1条延误", "status": "warning"},
```

**发现异常：** 校车A2线延误，点击查看（F-BUS-001）：


#### 🕘 09:30 — 人工录入集体迟到记录（F-ATT-001）

**场景：** BUS-A2校车延误，1C班12名学生集体迟到，需紧急人工录入。

```
[Step 1] 进入「出勤管理」→「人工录入」→ 选择班级：1C，日期：2026-05-27
```

### 3.1 跨角色共性问题（影响2个或以上角色）

#### 🔴 共性问题1：家长通知触达率保障 [高优先]

| 涉及角色 | 问题表现 |
|----------|----------|
| 校务主任（陈主任）| 校车延误时一键通知家长，但无法确认家长是否收到 |
| 校务处同工（李同工）| 临时变更午膳需逐个打电话通知家长，无高效渠道 |
| 教师（王老师）| 学生异常情况无法主动联系家长，只能等家长联系 |

**根本原因：** 当前微信渠道token状态显示"unknown"（见openclaw status），消息推送可靠性无法保证。

**建议方案：**
1. 建立消息送达回执机制（家长确认收到 → 系统标记"已读"）
2. 对高优先级通知（紧急/迟到/安全）增加短信备用渠道
3. 每24小时自动检查微信token有效性，异常时发送告警至校务主任

---

#### 🔴 共性问题2：家长与学校之间的通讯断层 [高优先]

| 涉及角色 | 问题表现 |
|----------|----------|
| 校务处同工（李同工）| 家长WhatsApp/微信联系临时变更，需手动录入系统 |
| 教师（王老师）| 无法在系统内以老师身份联系家长 |
| 家长（黄太太）| 申请被拒时找不到学校联系方式 |

**根本原因：** 家长渠道（微信/WhatsApp）与学校内部系统之间缺乏双向同步机制。

**建议方案：**
1. 建立「家长自助服务门户」（微信小程序），覆盖请假、午膳变更、费用查询
2. 教师端增加「发送消息给家长」功能（系统内消息，老师和家长的对话历史可查）
3. 所有"申请被拒"或"需联系学校"的场景，必须内嵌快捷联系方式（致电 + 发消息按钮）

---

#### 🟡 共性问题3：eClass/WebSAMS数据同步兜底机制 [高优先]

| 涉及角色 | 问题表现 |
|----------|----------|
| 校务处同工（李同工）| "部分成功"状态无法快速识别受影响学生 |
| 教师（王老师）| 出勤数据来源不可靠，影响班级出勤统计准确性 |
| 校务主任（陈主任）| 无法实时掌握数据同步健康状态 |

**根本原因：** 系统规格定义了数据同步策略（F-INT-001/F-INT-002），但未明确同步失败时的人工兜底界面和流程。

**建议方案：**
1. 每台设备/每个数据源单独显示状态（不合并为"部分成功"）
2. 同步失败时自动生成「受影响学生列表」
3. 提供一键「标记全勤」功能（当所有来源均失败时，由校务主任授权操作）

#### 🟡 共性问题4：敏感数据的界面展示策略 [高优先]

| 涉及角色 | 问题表现 |
|----------|----------|
| 校务主任（陈主任）| 财务审批页面显示完整身份证号 |
| 教师（王老师）| 学生详情页字段受限，但家长无法看到任何联系方式 |

**根本原因：** RBAC定义了访问权限，但未定义「允许访问后，界面如何展示」的策略。

**建议方案：**
1. 身份证号：默认显示为 A123456(X)，点击「查看」需二次确认（密码或OTP）
2. 电话号码：显示前3后4位（如 912***567），点击「查看」需二次确认
3. 完整地址：完全隐藏，仅在紧急情况（如学生安全）下由校务主任授权查看
4. 所有查看动作均记录至审计日志（F-USER-005）
