### 1.2 调用模式选择原则

| 模式 | 适用场景 | 示例 |
|------|---------|------|
| **直接依赖注入** | 同一进程内同步调用，强一致性要求 | LeaveModule → AuditModule |
| **EventEmitter** | 同一进程内异步通知，无需等待结果 | Leave审批通过 → 通知Module |
| **Kafka 消息** | 跨服务异步通信，最终一致性 | 请假审批 → 通知Module / 第三方系统 |

## 2. 内部服务调用规范

### 2.1 NestJS Module 间直接依赖注入

#### 规范要求

1. **接口隔离原则**：调用方通过 `Service` 类注入，而非直接依赖具体实现
2. **禁止循环依赖**：使用 `forwardRef()` 解决，需明确注释
3. **事务边界**：涉及多表写入的操作需在调用方开启事务

#### 标准实现模式

```typescript
// user.service.ts
@Injectable()
export class UserService {
  constructor(
    private readonly userRepository: UserRepository,
    private readonly auditService: AuditService,  // 内部模块注入
  ) {}

  async createUser(dto: CreateUserDto, actorId: string): Promise<User> {
    const user = await this.userRepository.save(dto);
    
    // 内部调用（同步，带审计）
    await this.auditService.log({
      action: 'user.create',
      userId: actorId,
      targetId: user.id,
      targetType: 'user',
      metadata: { name: user.name },
    });
    
    return user;
  }
}
```

#### Module 导出规范

```typescript
// user.module.ts
@Module({
  imports: [TypeOrmModule.forFeature([User]), AuditModule],
  controllers: [UserController],
  providers: [UserService],
  exports: [UserService],  // ← 必须显式导出
})
export class UserModule {}
```

### 2.2 事件驱动调用（EventEmitter）

#### 规范要求

1. **事件命名**：使用 `DomainEvent` 后缀，如 `UserCreatedEvent`
2. **Payload 结构**：统一 `{ eventType, aggregateId, payload, occurredAt }`
3. **禁止在事件处理器中抛出异常**：使用 try-catch 包裹并记录

#### 事件基类

```typescript
// common/events/base.event.ts
export abstract class DomainEvent {
  abstract readonly eventType: string;
  readonly occurredAt: Date;

  constructor(
    public readonly aggregateId: string,
    public readonly payload: Record<string, any>,
  ) {
    this.occurredAt = new Date();
  }
}
```

#### 事件发射器使用模式

```typescript
// leave.service.ts
@Injectable()
export class LeaveService {
  constructor(
    private readonly eventEmitter: EventEmitter2,
  ) {}

  async approve(id: string, dto: ApproveLeaveDto, actorId: string) {
    const leave = await this.leaveRepository.findById(id);
    leave.status = 'approved';
    leave.reviewedBy = actorId;
    await this.leaveRepository.save(leave);

    // 发射领域事件（异步通知）
    this.eventEmitter.emit(
      'leave.approved',  // 事件名
      new LeaveApprovedEvent(leave.id, {   // Payload
        studentId: leave.studentId,
        leaveType: leave.leaveType,
        startDate: leave.startDate,
        endDate: leave.endDate,
        approvedBy: actorId,
      }),
    );
  }
}
```

#### 事件监听器

```typescript
// notification.service.ts
@Injectable()
export class NotificationService {
  @OnEvent('leave.approved')
  async handleLeaveApproved(event: LeaveApprovedEvent) {
    try {
      // 发送通知给班主任、任课教师
      await this.sendLeaveApprovalNotification(event.payload);
    } catch (error) {
      // 记录但不抛出，防止事件总线中断
      this.logger.error(`Failed to send leave notification: ${error.message}`);
    }
  }

  @OnEvent('leave.rejected')
  async handleLeaveRejected(event: LeaveRejectedEvent) {
    // 发送拒绝通知
    await this.sendLeaveRejectionNotification(event.payload);
  }
}
```

### 2.3 消息队列调用（Kafka）

#### 规范要求

1. **Topic 命名**：`{domain}.{entity}.{action}`，如 `user.created`、`leave.approved`
2. **消息格式**：统一 JSON，携带 Schema 版本号
3. **幂等性**：消费者必须实现幂等处理（基于 `messageId` 去重）
4. **死信队列**：消费失败超过 3 次的消息进入 DLQ

#### Kafka 消息格式

```typescript
interface KafkaMessage<T> {
  messageId: string;       // UUID v4，幂等性 Key
  version: string;          // Schema 版本，如 '1.0.0'
  eventType: string;        // 事件类型
  aggregateId: string;      // 聚合根 ID
  timestamp: string;         // ISO 8601 时间戳
  source: string;           // 来源服务，如 'school-admin-backend'
  payload: T;               // 业务负载
}
```

#### 生产者实现

```typescript
// leave.event-producer.ts
@Injectable()
export class LeaveEventProducer {
  constructor(private readonly kafkaService: KafkaService) {}

  async publishLeaveApproved(leave: LeaveApplication) {
    await this.kafkaService.send('leave.approved', {
      messageId: uuidv4(),
      version: '1.0.0',
      eventType: 'leave.approved',
      aggregateId: leave.id,
      timestamp: new Date().toISOString(),
      source: 'school-admin-backend',
      payload: {
        applicationNo: leave.applicationNo,
        studentId: leave.studentId,
        classId: leave.classId,
        leaveType: leave.leaveType,
        startDate: leave.startDate,
        endDate: leave.endDate,
        approvedBy: leave.reviewedBy,
      },
    });
  }
}
```

#### 消费者实现

```typescript
// notification.leave-consumer.ts
@Injectable()
export class LeaveNotificationConsumer {
  @Process('leave.approved')
  async handleLeaveApproved(message: ConsumeMessage) {
    const data = JSON.parse(message.value.toString()) as KafkaMessage<LeaveApprovedPayload>;
    
    // 幂等性检查
    const processed = await this.checkProcessed(data.messageId);
    if (processed) {
      this.logger.info(`Message ${data.messageId} already processed, skipping`);
      return;
    }

    try {
      await this.notificationService.sendLeaveApprovalNotification(data.payload);
      await this.markProcessed(data.messageId);
    } catch (error) {
      if (data.retryCount >= 3) {
        // 进入死信队列
        await this.kafkaService.sendToDLQ('leave.approved.dlq', data);
      }
      throw error;  // 触发重试
    }
  }
}
```

| `@IsArray()` | 数组类型 | ids[] |
| `@ValidateNested()` | 嵌套对象验证 | nested object |

## 5. 数据库事务规范

### 5.1 事务边界定义

#### 单一聚合根操作（无事务）

```typescript
// 简单单表操作不需要显式事务
async createLeave(dto: CreateLeaveDto): Promise<LeaveApplication> {
  const leave = this.leaveRepository.create(dto);
  return this.leaveRepository.save(leave);
}
```

#### 本地事务（同一数据库）

```typescript
// 需要多表写入时使用事务装饰器
@Transaction()
async createLeaveAndNotify(
  dto: CreateLeaveDto,
  actorId: string,
): Promise<LeaveApplication> {
  // Step 1: 创建请假记录
  const leave = this.leaveRepository.create(dto);
  await this.leaveRepository.save(leave);

  // Step 2: 更新学生出勤（生成 excused 记录）
  await this.attendanceService.markExcused(
    leave.studentId,
    leave.startDate,
    leave.endDate,
  );

  // Step 3: 发送通知
  await this.notificationService.sendLeaveApplicationNotice(leave);

  return leave;
}
```

#### 分布式事务（Saga 模式）

对于跨数据库/跨服务的操作，采用 Saga 编排模式：

```
请假审批 Saga：
┌────────────┐    ┌──────────────┐    ┌──────────────┐
│ 请假服务    │───▶│ 更新请假状态  │───▶│ 通知服务      │
│ (Saga Start)│    │ (本地事务)   │    │ (异步补偿)   │
└────────────┘    └──────┬───────┘    └──────────────┘
                        │
                        ▼
               ┌──────────────────┐
               │ 补偿事件:        │
               │ leave.rollback  │
               │ (失败时回滚状态) │
               └──────────────────┘
```

```typescript
// LeaveSaga.ts
@Injectable()
export class LeaveSaga {
  async executeLeaveApprovalSaga(leaveId: string, actorId: string) {
    const sagaId = uuidv4();
    
    try {
      // Step 1: 审批请假（本地事务）
      const leave = await this.transactionalOperation(async () => {
        return this.approveLeaveWithTransaction(leaveId, actorId);
      });

      // Step 2: 更新出勤记录（发布事件，异步）
      await this.eventEmitter.emit('leave.approved', {
        sagaId,
        leaveId: leave.id,
        studentId: leave.studentId,
        startDate: leave.startDate,
        endDate: leave.endDate,
      });

      // Step 3: 发送通知（发布事件，异步）
      await this.eventEmitter.emit('notification.leave_approved', {
        sagaId,
        leaveId: leave.id,
        studentId: leave.studentId,
      });

    } catch (error) {
      // Saga 补偿：发布补偿事件
      await this.eventEmitter.emit('leave.approval_failed', {
        sagaId,
        leaveId,
        error: error.message,
        compensatedAt: new Date().toISOString(),
      });
      throw error;
    }
  }
}
```

### 5.2 事务隔离级别

| 隔离级别 | 使用场景 | 代价 |
|----------|---------|------|
| `READ COMMITTED`（默认）| 绝大多数操作 | 适中 |
| `READ UNCOMMITTED` | 只读查询 | 低 |
| `REPEATABLE READ` | 财务交易（金额计算）| 高 |
| `SERIALIZABLE` | 强一致性要求的配置更新 | 最高 |

```typescript
// 财务金额计算使用 SERIALIZABLE
@Transaction(TransactionIsolationLevel.SERIALIZABLE)
async processPayment(dto: PaymentDto): Promise<void> {
  // 确保并发金额计算一致性
  const current = await this.accountRepository.getBalance(dto.studentId);
  await this.accountRepository.updateBalance(dto.studentId, current + dto.amount);
}
```

### 5.3 事务回滚规范

```typescript
// 正确：显式抛出特定异常
async createLeave(dto: CreateLeaveDto) {
  try {
    const existing = await this.leaveRepository.findPending(dto.studentId, dto.startDate);
    if (existing) {
      throw new BusinessException('LEAVE-002');  // 业务异常
    }
    return await this.leaveRepository.save(this.leaveRepository.create(dto));
  } catch (error) {
    if (error instanceof BusinessException) throw error;
    throw new SystemException('DATABASE_ERROR', error);  // 系统异常
  }
}

// 错误：在事务中吞掉异常
async createLeave(dto: CreateLeaveDto) {
  await this.transactionalOperation(async () => {
    try {
      return await this.leaveRepository.save(dto);
    } catch (error) {
      this.logger.error(error);  // ❌ 错误：吞掉异常，事务不会回滚
    }
  });
}
```

### 5.4 审计日志与事务

所有数据变更必须在同一事务中写入审计日志：

```typescript
@Transaction()
async approveLeave(id: string, dto: ApproveLeaveDto, actorId: string) {
  // 1. 更新请假状态
  await this.leaveRepository.update(id, {
    status: 'approved',
    reviewedBy: actorId,
    reviewedAt: new Date(),
  });

  // 2. 同步写入审计日志（同一事务）
  await this.auditRepository.save({
    action: 'leave.approve',
    userId: actorId,
    resourceType: 'leave_application',
    resourceId: id,
    changes: { before: { status: 'pending' }, after: { status: 'approved' } },
    ipAddress: getFromContext(),
  });
}
```

### 5.5 事务传播行为

| 传播行为 | 说明 | 使用场景 |
|----------|------|---------|
| `REQUIRED`（默认）| 有则加入，无则创建新事务 | 绝大多数操作 |
| `REQUIRES_NEW` | 始终创建新事务 | 日志写入（独立于业务事务）|
| `NESTED` | 嵌套事务（Savepoint）| 子操作失败不影响父事务 |
| `MANDATORY` | 必须在已有事务中执行 | 安全敏感的共享操作 |
| `NEVER` | 禁止在事务中执行 | 外部 API 调用前检查 |
