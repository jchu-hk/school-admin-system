## 1. 运维组织与职责

### 1.1 运维角色定义

| 角色 | 职责范围 | 技能要求 |
|------|----------|----------|
| **校务主任 (SRO)** | 业务连续性最终责任人；灾难恢复决策；跨部门协调 | 系统整体业务理解 |

## 2. 日常运维清单

### 2.1 每日任务（每日 08:30 前完成）

#### 系统健康检查

| # | 检查项 | 检查方法 | 预期结果 | 异常处理 |
|---|--------|----------|----------|----------|
| D-01 | **数据库连接** | `pg_stat_activity` 连接数；PgBouncer 状态 | 连接成功率 ≥ 99%；连接池使用率 < 70% | 见 Runbook R-01 |
| D-02 | **Redis 服务** | Redis CLI `info clients`；`cluster info` | 客户端连接数正常；无过期 key 堆积 | 见 Runbook R-03 |
| D-03 | **API 服务可用性** | Prometheus `up` 指标（各 service） | 所有 service up=1 | 重启对应 Deployment |
| D-04 | **磁盘空间** | `df -h` 各挂载点 | 所有分区 < 85% 使用率 | 清理日志或扩容 |
| D-05 | **内存使用** | `kubectl top nodes` / `free -h` | 各服务内存 < 90% 使用率 | 重启或扩容 |
| D-06 | **Kafka 消费延迟** | `kafka-consumer-groups --describe` | consumer lag < 1000 messages | 扩展消费者 |
| D-07 | **备份任务执行** | 检查 `backup_jobs` 表最后执行时间 | 每日增量备份 01:00 成功 | 手动触发并告警 |

#### 业务数据检查

| # | 检查项 | 检查方法 | 预期结果 | 异常处理 |
|---|--------|----------|----------|----------|
| D-08 | **eClass 数据同步** | 检查 `eclass_sync_logs` 最近记录 | 最后同步时间 < 24h；status=success | 检查 API Token；重试同步 |
| D-09 | **WebSAMS 数据同步** | 检查 `websams_sync_logs` | 最后同步时间正常；无 pending 任务 | 联系 EDB 技术支持 |

### 4.3 恢复流程

#### 4.3.1 数据库 PITR（时间点恢复）

```
前提：PostgreSQL 数据损坏或误删
目标 RPO：≤ 1 小时（WAL 归档）
预计耗时：15-30 分钟

Step 1: 停止写入服务
  kubectl scale deployment dashboard-deploy cyclic-deploy --replicas=0

Step 2: 记录恢复时间点
  SELECT now();

Step 3: 找到最近全量备份
  检查 S3: s3://school-admin-backups/postgres/daily/

Step 4: 执行 PITR 恢复
  docker run --rm postgres:16 pg_restore --verbose --clean --if-exists \
    --dbname=postgres s3://school-admin-backups/postgres/daily/YYYY-MM-DD-full.dump

Step 5: 应用 WAL 至指定时间点
  pg_restore ... --wal-seek=TIMESTAMP

Step 6: 数据完整性校验
  执行 verify-backup-integrity.sh（MD5 + 记录数核对）

Step 7: 重启服务
  kubectl scale deployment dashboard-deploy cyclic-deploy --replicas=3

Step 8: 业务验证（考勤数据今日、财务记录完整性）

Step 9: 通知
  Slack #ops-alerts + SMS 至校务主任
```

#### 4.3.2 文件级恢复

```
前提：MinIO 中文件误删
目标 RTO：≤ 15 分钟

Step 1: 确定误删文件范围（Bucket + Prefix + 时间）
Step 2: 从 S3 版本控制恢复
  mc version info <alias>/<bucket>
  mc cp --version-id <vid> <src> <dest>
Step 3: 验证文件 MD5
Step 4: 通知相关业务模块负责人
```

## 5. 灾难恢复步骤（F-NEW-001）

> 对应 SPEC-COMPLETE.md v1.6.0 — F-NEW-001 一键灾难恢复与业务连续性

### 5.1 灾难恢复目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **RTO（恢复时间目标）** | ≤ 4 小时 | 系统从故障到恢复可用的最长时间 |
| **RPO（恢复点目标）** | ≤ 1 小时 | 数据丢失的最大时间窗口 |
| **备份成功率** | ≥ 99.9% | 每日备份任务成功率 |
| **恢复演练频率** | 每季度 1 次 | 季度灾难恢复演练验证 |
| **恢复验证时间** | ≤ 30 分钟 | 恢复后数据完整性验证时间 |

### 5.2 灾难场景分级与响应

| 灾难等级 | 场景描述 | RTO | RPO | 响应时间 |
|----------|----------|-----|-----|----------|
| **L1 - 数据层** | PostgreSQL 损坏、误删表 | 1h | 15min | 立即 |
| **L2 - 应用层** | 应用崩溃、配置错误 | 2h | 0（无数据丢失）| 15min |
| **L3 - 系统层** | 整机故障、硬件损坏 | 4h | 1h | 立即 |
| **L4 - 网络层** | 网络攻击、线路中断 | 2h | 0 | 立即 |

### 5.3 一键灾难恢复执行流程

```
一键恢复触发条件
  - 自动：监控系统检测 Critical 告警持续 > 5 分钟
  - 手动：运维工程师或校务主任执行

一键恢复命令
  kubectl exec -it disaster-recovery-pod -- /scripts/disaster-recovery.sh --mode=full

执行流程

Step 1: 故障自动评估（自动）
  - 系统自动检测故障类型（L1/L2/L3/L4）
  - 选择最优恢复策略，展示恢复计划预览

Step 2: 备份完整性验证（自动）
  - MD5 校验
  - WAL 连续性检查
  - 显示可用恢复点时间戳

Step 3: 数据层恢复（自动）
  - PostgreSQL PITR
  - Redis 快照恢复
  - MinIO 文件恢复
  - 实时显示进度条

Step 4: 应用层恢复（自动）
  - Docker 镜像拉取
  - 配置注入（ConfigMap + Vault）
  - Service 启动

Step 5: 数据完整性校验（自动）
  - 记录数抽样核对
  - 业务数据逻辑校验
  - 关键表数据存在性验证

Step 6: 服务可用性探测（自动）
  - HTTP 健康检查
  - API 端到端测试
  - 数据库读写验证

Step 7: 恢复完成通知（自动）
  - SMS + App 通知校务主任
  - Slack #ops-alerts 公告
  - 生成恢复报告存档至 F-USER-005

预计总耗时
  - L1 数据层：15-30 分钟
  - L2 应用层：5-15 分钟
  - L3 系统层：2-4 小时
  - L4 网络层：1-2 小时
```

### 5.4 DR 演练计划（每季度）

## 7. 应急响应流程

### 7.1 事件分级

| 级别 | 定义 | 响应时间 | 升级路径 |
|------|------|----------|----------|
| **P0 — 紧急** | 核心服务全面不可用，影响全校使用 | 立即（0 分钟）| 运维 → 校务主任 → 校长 |
| **P1 — 严重** | 核心服务部分不可用，特定功能故障 | 15 分钟 | 运维 → 系统管理员 |
| **P2 — 中等** | 非核心功能故障，有 workaround | 1 小时 | 运维工程师处理 |
| **P3 — 轻微** | UI 问题、性能轻微下降 | 下一个工作日 | 记录并排期修复 |

### 7.2 应急响应标准流程

```
P0/P1 事件响应流程

T+0min: 发现/报告事件
  - 告警触发 / 用户报告 / 值班工程师发现
  - 在 #incidents Slack 频道创建事件记录
  - 指派 Incident Commander（IC）

T+5min: 初步评估
  - IC 评估影响范围和级别
  - 决定是否启动备用方案（降级服务）
  - 如 P0：立即通知校务主任

T+15min: 根因定位
  - 查阅 Grafana / Loki / Tempo
  - 检查最近部署变更（GitOps 历史）
  - 如不确定原因：回滚最近一次部署

T+30min: 修复执行
  - 执行对应 Runbook
  - 或执行回滚操作（kubectl rollout undo）
  - 持续更新 #incidents 进展

T+1h: 服务恢复验证
  - 确认所有健康检查通过
  - 通知校务主任「服务已恢复」
  - 在 #incidents 发布事后报告

T+24h: 事件报告
  - 编写 Incident Report（RCA）
  - 分析根因，制定预防措施
  - 更新 Runbook（如需要）
  - 归档至运维文档库
```

## 8. 运维Runbook手册

### R-01: 数据库连接池耗尽

**症状：** 应用报错 `remaining connection slots are reserved...`；PgBouncer 显示 `pool_used > pool_max`

**诊断步骤：**
```bash
# 1. 检查 PgBouncer 连接池状态
psql -h pgbouncer -p 6432 -U pgbouncer -c "show pools"

# 2. 检查 PostgreSQL 当前连接数
psql -h postgres-primary -U app_user -c "SELECT count(*) FROM pg_stat_activity WHERE state='active'"

# 3. 检查是否有长事务
psql -h postgres-primary -U app_user -c \
  "SELECT pid, usename, query_start, state, query FROM pg_stat_activity WHERE state != 'idle' ORDER BY query_start"
```

**修复步骤：**
```bash
# 方案1: 终止长时间运行的查询
psql -h postgres-primary -U admin -c "SELECT pg_terminate_backend(<pid>)"

# 方案2: 重启应用（解决连接泄漏）
kubectl rollout restart deployment dashboard-deploy

# 方案3: 扩容 PgBouncer 连接池（临时）
kubectl patch deployment pgbouncer-deploy -p '{"spec":{"replicas":5}}'

# 方案4: 扩容 PostgreSQL 最大连接数（临时）
kubectl exec -it postgres-primary -- psql -U postgres -c "ALTER SYSTEM SET max_connections=200"
kubectl rollout restart statefulset postgres-primary
```

**预防措施：** 设置 `idle_in_transaction_session_timeout = 30min`；监控慢查询并优化

### R-03: Redis 集群故障

**症状：** Redis 连接报错 `CLUSTERDOWN`；缓存服务不可用

**诊断步骤：**
```bash
# 1. 检查 Redis Cluster 状态
redis-cli -h redis-cluster.cluster.local -p 6379 cluster info

# 2. 检查各节点状态
redis-cli -h redis-node-1.cluster.local cluster nodes

# 3. 检查 Sentinel 状态
redis-cli -h redis-sentinel.cluster.local sentinel master mymaster
```

**修复步骤：**
```bash
# 方案1: 单节点故障，触发自动故障转移
redis-cli -h redis-node-1.cluster.local cluster failover takeover

# 方案2: 从 RDB 快照恢复
kubectl exec -it redis-backup-job -- /scripts/restore-redis.sh

# 方案3: 重建集群
redis-cli --cluster-recreate --cluster-yes <nodes-list>
```

**预防措施：** 确保 Redis Cluster 至少 3 主 3 从；监控 `redis_connected_clients` 指标

### R-04: Kafka 消费滞后

**症状：** consumer lag 持续增长；消息处理延迟 > 10 分钟

**诊断步骤：**
```bash
# 1. 检查消费者组 lag
kafka-consumer-groups --bootstrap-server kafka:9092 \
  --group school-admin-consumer --describe

# 2. 检查消费者进程状态
kubectl get pods -l app=kafka-consumer
kubectl logs -l app=kafka-consumer --tail=50
```

**修复步骤：**
```bash
# 方案1: 扩容消费者数量
kubectl scale deployment ai-deploy --replicas=4

# 方案2: 手动重置 offset 到最新（跳过积压数据）
kafka-consumer-groups --bootstrap-server kafka:9092 \
  --group school-admin-consumer --reset-offsets \
  --topic school-admin-events --to-latest --execute

# 方案3: 确认无积压后恢复消费者
kubectl rollout restart deployment ai-deploy
```

### R-05: 证书过期

**症状：** 告警 `certificate will expire in N days`；HTTPS 连接报错

**诊断步骤：**
```bash
# 1. 检查证书到期时间
openssl s_client -connect api.school.edu.hk:443 -servername api.school.edu.hk 2>/dev/null | openssl x509 -noout -dates

# 2. 检查 Let's Encrypt 证书状态
certbot certificates

# 3. 检查 Vault 证书管理
vault list secret/certificates
```

**修复步骤：**
```bash
# 方案1: Let's Encrypt 自动续期
certbot renew --quiet

# 方案2: 手动更新（Vault 管理）
vault write secret/certificates/api-ssl \
  cert=@/path/to/new-cert.pem key=@/path/to/new-key.pem

# 方案3: 重新签发
certbot certonly --nginx -d api.school.edu.hk
```

### R-06: 服务崩溃（HPA 未响应）

**症状：** Pod 持续 CrashLoopBackOff；HPA 未自动扩容

**诊断步骤：**
```bash
# 1. 检查 Pod 状态
kubectl get pods -l app=dashboard

# 2. 查看崩溃日志
kubectl logs <pod-name> --previous

# 3. 检查资源限制
kubectl describe pod <pod-name> | grep -A 5 "Limits\|Requests"

# 4. 检查 HPA 状态
kubectl get hpa -l app=dashboard
```

**修复步骤：**
```bash
# 方案1: 如果资源不足，增大限制
kubectl patch deployment dashboard-deploy -p '{"spec":{"template":{"spec":{"containers":[{"name":"dashboard","resources":{"limits":{"memory":"2Gi"},"requests":{"memory":"512Mi"}}}]}}}}'

# 方案2: 如果是配置错误，检查环境变量
kubectl exec -it <pod-name> -- env | grep -E "DB_|REDIS_|KAFKA_"

# 方案3: 如果无法修复，手动扩容（绕过 HPA）
kubectl scale deployment dashboard-deploy --replicas=5

# 方案4: 查看详细错误
kubectl describe pod <pod-name> | tail -30
```
